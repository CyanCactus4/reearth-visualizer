# Для того, кто только заходит в проект

Привет. Если ты человек или агент и тебе нужно не тонуть в первый день — прочитай это целиком один раз. Это не дублирование официального README построчно, а то, что я хотел бы знать **до** того, как начал ковырять код.

## С чего начать (порядок имеет значение)

1. **Прочитай корневой** [README.md](README.md) — там честная картинка: Docker для бэкенда, `web/` для фронта, переменные `.env` для Cesium и API.
2. **Открой** [server/AGENTS.md](server/AGENTS.md). Там структура Go-модуля, команды `make`, стиль коммитов, где лежит GraphQL. Это короткий и точный ориентир.
3. **Если работаешь из Cursor или другого агента** — загляни в [COMMANDS.md](COMMANDS.md): там список команд, которые разумно разрешать в allowlist, чтобы не спотыкаться об каждый `git commit` и `make test`.
4. **Когда нужна глубина** — [DETAILS.md](DETAILS.md) в корне: там длинное описание архитектуры, стека, тестов и файлов (особенно коллаборации). Не обязательно читать за раз; используй как справочник.

## Что это за репозиторий одной фразой

**Re:Earth Visualizer** — веб-приложение для GIS: редактор сцен, слои, сторителлинг, плагины, публикация. Бэкенд на **Go**, фронт на **React + TypeScript + Vite**, данные в **MongoDB**, файлы в **GCS/S3** (локально — fake GCS). Есть **реалтайм-коллаборация** (WebSocket, при необходимости Redis).

## Локальный стенд без сюрпризов

- **Docker Compose лежит в корне репозитория**, не только в `server/`. Тома для Mongo и GCS привязаны к `./tmp/mongo` и т.д. **относительно файла compose**. Если запускаешь compose из `server/`, пути всё равно должны резолвиться правильно — это уже учтено в репозитории; не создавай дублирующие «левые» каталоги с root-only правами без нужды.
- После изменений compose для Mongo перезапусти контейнер БД, иначе старый процесс останется со старыми лимитами.
- Если **`make test` падает** с «too many open files» или Mongo внезапно падает посреди прогона — см. раздел про тесты в DETAILS.md: там про `TEST_P`, `test-no-e2e` и ulimits. На слабой машине это не «баг тестов», а нагрузка.

## Команды, которые реально используются каждый день

Из корня:

- Бэкенд-сервисы: в документации обычно `cd server && make d-run`, потом один раз `make setup-dev`.
- Фронт: `cd web && yarn && yarn start` при наличии `.env` (см. README).

Тесты:

- Сервер: `cd server && make test` (нужен доступный Mongo на `REEARTH_DB`, по умолчанию `mongodb://localhost`).
- Фронт: `cd web && yarn test`, строже — `yarn type` и `yarn lint`.
- Тяжёлый e2e на Go: `cd server && make e2e`. Playwright живёт отдельно в `e2e/`.

## Git и артефакты

- **Не коммить** собранный бинарник `server/server`, если он случайно появился после `go build`.
- Коммиты в этом проекте ожидаются в духе **Conventional Commits** со scope, как в server/AGENTS.md (`feat(server): …`, `fix(web): …`).
- Если добавляешь большой снимок работы — лучше одно осмысленное сообщение с телом, чем десять «wip».

## Где искать «логику коллаборации»

- **Сервер:** почти всё под `server/internal/collab/`. Точка входа WebSocket — `ws.go`, маршрутизация комнат — `hub.go`.
- **Клиент:** `web/src/services/collab/` (протокол, очередь офлайна, провайдер) и компоненты редактора в `web/src/app/features/Editor/` с префиксом `Collab*`.
- **GraphQL:** схемы кусками в `server/gql/*.graphql`, резолверы в `server/internal/adapter/gql/`.

## Когда застрял

1. Проверь, что Mongo и (если нужно) GCS/reearth-accounts подняты так, как описано в README.
2. Прогон с урезанным параллелизмом: `make test TEST_P=2` из `server/`.
3. Поиск по репозиторию по строкам `collab`, `ServeWS`, `CollabProvider`.
4. Не стесняйся открыть DETAILS.md и найти нужный подзаголовок — он специально раздутый, чтобы не гадать.

## Тон

Кодовая база большая, зрелая, с историей. Новые изменения лучше делать **узкими диффами**, в стиле соседних файлов, с тестами там, где уже принято тестировать (Go — `*_test.go`, web — Vitest рядом с модулями). Если что-то кажется «очевидно неправильным» — возможно, это legacy или обходной путь под прод; тогда сначала найди комментарий или тест, объясняющий зачем.

Удачи. Если ведёшь дневник решений для команды — короткая запись в PR ценнее любого нового markdown-файла в корне без запроса.
