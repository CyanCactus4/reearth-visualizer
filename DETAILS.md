# Re:Earth Visualizer — подробное описание проекта

Этот документ намеренно **длинный**: в нём собраны суть продукта, структура репозитория, архитектурные слои, ориентир по коду (в первую очередь по подсистеме коллаборации и смежным файлам), стек технологий с плюсами и минусами, а также то, **как устроены и как запускаются тесты**. Отдельный раздел описывает **что было реализовано** в ходе недавней доработки реалтайм-редактирования. Текст рассчитан на читателя, который открывает репозиторий впервые и хочет понять «где что лежит» без догадок.

---

## 1. Суть продукта

**Re:Earth Visualizer** — это полнофункциональный **WebGIS-редактор** и платформа публикации картографических проектов. Пользователь работает в браузере: собирает сцену (слои, стили, виджеты, сторителлинг), подключает данные, настраивает свойства через динамическую систему **Property** (схемы и значения), публикует проект. Рендер глобуса/карты опирается на **Cesium** (на стороне клиента нужен Ion token или свой источник тайлов — см. README).

Серверная часть отвечает за **персистентность** (MongoDB), **авторизацию** (в т.ч. интеграция с Re:Earth Accounts, OIDC), **хранение файлов** (Google Cloud Storage или S3), **GraphQL API** и **дополнительные HTTP/gRPC** поверхности. Клиентская часть — **SPA** на React, общающаяся с API и WebSocket.

**Коллаборация** — это возможность **нескольким пользователям одновременно** находиться в одном проекте редактора: видеть присутствие, курсоры, активность, чат, блокировки ресурсов, применять изменения сцены через единый операционный слой с аудитом и откатом, не ломая консистентность Mongo.

## 3. Архитектура сервера (слои и поток данных)

Сервер следует привычной для Go-сервисов схеме **чистых границ** (не строгий DDD, но разделение есть):

1. **Транспортный слой** — `internal/adapter/`: GraphQL (`gql`), HTTP (`http`), middleware, интеграция с внешним Accounts API, внутренний gRPC API (`internalapi`). Адаптеры переводят входящие запросы в вызовы use case и обратно мапят ошибки/модели.
2. **Прикладной слой** — `internal/usecase/`: **interactor** реализуют сценарии («создать проект», «обновить слой»), **repo** — интерфейсы к хранилищам, **gateway** — к внешним сервисам (файлы, политика).
3. **Инфраструктура** — `internal/infrastructure/`: конкретные реализации репозиториев на **MongoDB**, in-memory заглушки для тестов, **GCS/S3**, файловая система, Auth0 и т.д.
4. **Сборка приложения** — `internal/app/`: конфигурация (`config`), wiring зависимостей, регистрация маршрутов Echo, GraphQL-сервер, health, трейсинг.
5. **Коллаборация** — `internal/collab/`: отдельный пакет с WebSocket-хабом, ретрансляцией через Redis, применением операций к сцене, чатом, курсорами, блокировками. Он связан с interactors и Mongo, но логически изолирован как подсистема.

**Поток типичного GraphQL-запроса:** HTTP → Echo → GraphQL handler → resolver в `adapter/gql` → interactor → repo (Mongo) → ответ. **Поток коллаборации:** HTTP upgrade → `collab.ServeWS` → регистрация `Conn` в комнате `Hub` по `projectId` → обмен JSON-сообщениями → при необходимости запись в Mongo через существующие interactors/репозитории и рассылка событий другим соединениям (и другим инстансам через Redis).

## 4. Пакет `server/internal/collab` — назначение каждого файла

Ниже перечислены **все** Go-файлы пакета `collab` (на момент составления документа). Для каждого указано **зачем файл существует**. Названия файлов — это не декорация: в Go принято дробить логику по темам, чтобы `go test` и навигация в IDE оставались простыми.

### 4.1. Транспорт WebSocket и соединение

- **`ws.go`** — HTTP-handler для Echo: **апгрейд** до WebSocket, лимиты размера сообщений и частоты, проверка `Origin`, привязка к `Hub`, чтение цикла сообщений, парсинг оболочки `ClientMessage` (`v`, `t`, `d`). Это **публичная точка входа** коллаборации для браузера.
- **`conn.go`** — обёртка над одним WebSocket-соединением: буфер исходящих сообщений, горутина writer, сериализация исходящих структур, метаданные пользователя/сессии. Хаб держит множество `Conn`.

### 4.2. Хаб, комнаты, снимки сцены

- **`hub.go`** — центральный **маршрутизатор**: комнаты по ID проекта, Redis-relay для мультиинстанса, таблица блокировок, лимитеры чата/курсоров/активности, подписчики на revision сцены, часы для LWW/HLC полей виджетов и свойств, стек операций, хранилище снапшотов. Это «мозг» реалтайма на одном процессе.
- **`hub_test.go`** — юнит- и интеграционно-ориентированные тесты хаба (подключение, рассылка, граничные случаи).
- **`hub_lock.go`** — часть логики хаба, связанная с **блокировками** (вынесена из монолита `hub.go` для читаемости).
- **`hub_scene_rev.go`** — учёт **версии/ревизии сцены**: уведомления клиентам при изменениях, связь с GraphQL/SSE при необходимости.
- **`scene_snapshot_hub.go`** — координация периодических или по-событию **снимков состояния сцены** (для восстановления, отладки или оптимизации).
- **`scene_snapshot_store.go`** — абстракция/реализация **хранения** снимков (куда пишем и как читаем).
- **`redis_relay.go`** — публикация/подписка в Redis, чтобы **несколько подов** сервера видели одни и те же комнаты.
- **`redis_scene_rev_test.go`** — тесты сценариев ревизии с Redis (или заглушками).

### 4.3. Диспетчеры событий (чат, курсоры, активность)

- **`chat_dispatch.go`** — разбор входящих сообщений чата, маршрутизация в хаб, связь с историей и модерацией.
- **`chat_dispatch_test.go`** — тесты диспетчера чата.
- **`cursor_dispatch.go`** — обработка **позиций курсора** удалённых пользователей (throttle, валидация, broadcast).
- **`cursor_dispatch_test.go`** — тесты курсоров.
- **`activity_dispatch.go`** — «мелкая» активность: печатает в чате, двигается по UI карты и т.п. (разные виды с отдельными лимитерами).
- **`activity_dispatch_test.go`** — тесты активности.

### 4.4. Чат: API, история, упоминания

- **`chat_api.go`** — HTTP-часть или вспомогательные методы для **чата** снаружи WebSocket (зависит от wiring в app).
- **`chat_api_test.go`** — тесты API чата.
- **`chat_history.go`** — хранение и выдача **истории** сообщений (интеграция с Mongo или in-memory для тестов).
- **`chat_mentions.go`** — разбор **@упоминаний** в тексте, извлечение получателей.
- **`chat_mentions_test.go`** — тесты упоминаний.
- **`mention_webhook.go`** — опциональная отправка уведомлений во внешний webhook при упоминании.

### 4.5. Блокировки ресурсов

- **`lock_dispatch.go`** — обработка сообщений **lock/unlock**, согласование с таблицей блокировок и Redis.
- **`lock_dispatch_resource_test.go`** — тесты блокировок по ресурсам.
- **`lock_table.go`** — **in-memory** таблица: кто держит какой ресурс, TTL, конфликты.
- **`lock_table_test.go`** — тесты таблицы.
- **`lock_redis.go`** — **распределённые** блокировки при наличии Redis-клиента.
- **`lock_redis_test.go`** — тесты Redis-блокировок.
- **`lock_holder_wire.go`** — сериализация/контракт «держатель блокировки» между клиентом и сервером (wire-формат).
- **`lock_holder_wire_test.go`** — тесты wire-формата.
- **`project_scene_access.go`** — проверки **доступа** к проекту/сцене перед чувствительными операциями коллаборации (связка с доменной моделью).

### 4.6. Применение операций к сцене (apply)

Ядро — **`apply.go`**: разбор типов операций, вызов специализированных обработчиков, единая обработка ошибок и версий. Остальные файлы с префиксом **`apply_`** — это **разбор по доменным областям** (слои NLS, стили, виджеты, сторителлинг, камера сцены, свойства и т.д.), чтобы файл не превратился в тысячи строк.

Примеры (имена говорят о предметной области):

- **`apply_merge_property_json.go`** — слияние JSON свойств с учётом **CAS/часов документа** (важно для конфликтующих правок).
- **`apply_property_value.go`**, **`apply_property_items.go`**, **`apply_property_value_undo.go`**, **`apply_property_value_decode_test.go`** — значения свойств, элементы, откат, тесты декодирования.
- **`apply_nls_layer.go`**, **`apply_nls_layer_undo.go`**, **`apply_nls_layer_undo_test.go`** — слои NLS и откаты.
- **`apply_nls_geojson_feature.go`**, **`apply_nls_geojson_undo.go`** — GeoJSON внутри слоя.
- **`apply_nls_infobox_block.go`**, **`apply_nls_photo_overlay.go`**, **`apply_nls_custom_properties.go`** — инфобоксы, фотооверлеи, кастомные поля.
- **`apply_style.go`**, **`apply_style_undo.go`**, **`apply_style_undo_test.go`** — стили.
- **`apply_widget_undo.go`**, **`apply_widget_undo_test.go`** — виджеты и откат.
- **`apply_story.go`**, **`apply_story_page.go`**, **`apply_story_page_undo.go`**, **`apply_story_page_undo_test.go`** — сторителлинг.
- **`apply_scene_camera.go`** — камера сцены.
- **`apply_merge_property_json_undo_test.go`**, **`apply_nls_layer_undo_test.go`** и другие файлы с суффиксом `*_undo_test.go` — регрессия откатов.

Вспомогательные к apply:

- **`apply_revision.go`** — нумерация/связь **ревизий** применения.
- **`apply_audit.go`**, **`apply_audit_api.go`**, **`apply_audit_api_test.go`** — **аудит** применённых операций (кто, когда, что), API для выборки.
- **`apply_test.go`** — общие тесты конвейера apply.
- **`apply_merge_property_json.go`** (уже выше) работает в паре с **`property_json_merge.go`**, **`property_json_merge_test.go`**.

### 4.7. Слияние, часы, HLC, стек операций

- **`property_field_clock.go`**, **`property_field_clock_test.go`** — **last-write-wins** часы на уровне поля свойства.
- **`property_field_hlc.go`** — **гибридные логические часы** для CRDT-подобного поведения полей.
- **`property_doc_clock.go`**, **`property_doc_clock_test.go`** — часы на уровне **целого документа** свойства (для merge JSON).
- **`hlc.go`**, **`hlc_test.go`** — реализация HLC и тесты.
- **`entity_clock.go`** — обобщённые часы сущностей, если нужно единообразие.
- **`property_json_merge.go`**, **`property_json_merge_test.go`** — алгоритм слияния JSON с учётом конфликтов.
- **`property_remove_item_undo.go`**, **`property_remove_item_undo_test.go`** — откат удаления элементов внутри свойства (item-level undo).

- **`op_stack.go`** — стек **операций** для undo/redo на сервере.
- **`undo_exec.go`** — выполнение шага отката.
- **`redo_forward_patch.go`**, **`redo_forward_patch_test.go`** — повторное применение (redo) через forward-patch.

### 4.8. HTTP для undo и админка

- **`undo_http.go`** — HTTP endpoints (или Echo handlers) для **undo/redo** с клиента.
- **`admin_restore_http.go`** — административные сценарии **восстановления** (осторожно: только для доверенных ролей; реализация зависит от конфигурации).

### 4.9. SSE, GraphQL-контекст, опции, фаззинг

- **`sse_scene_rev.go`** — **Server-Sent Events** как альтернативный/дополнительный канал обновления ревизии сцены (для клиентов без постоянного WS или для отладки).
- **`graphql_context.go`** — прокидывание контекста коллаборации в GraphQL (например user id, workspace).
- **`options.go`** — структура опций при создании хаба (TTL блокировок, лимиты, feature flags).

- **`collab_fuzz_test.go`** — **go fuzz** для устойчивости парсеров/merge к мусорным входам.

### 4.10. Сводка по количеству файлов

В каталоге `server/internal/collab` лежит **83** Go-файла (исходники и тесты вместе): это нормальный масштаб для подсистемы, где смешаны сетевой протокол, доменные операции и устойчивость к гонкам. Разделы 4.1–4.9 покрывают каждый из них по смыслу; дублировать здесь же полный алфавитный список не обязательно — его всегда можно получить командой `ls server/internal/collab/*.go`.

Если появятся новые файлы в `collab/`, их смысл обычно читается по префиксу: `apply_*` — доменная операция, `*_dispatch` — входящий тип сообщения, `*_test` — тесты.

## 5. Другие важные области сервера (кратко, но явно)

### 5.1. `internal/adapter/gql`

- Содержит **сгенерированные** резолверы и dataloaders (`go generate`), а также ручные резолверы `resolver_*.go`.
- **`resolver_collab_scene_rev.go`** — связь GraphQL с **ревизией сцены** для коллаборации (подписка/запрос актуального номера).
- **`resolver_mutation_featureCollection.go`**, **`resolver_mutation_nlslayer.go`** и др. — мутации домена; в коллаборации часто **дополняются** передачей контекста или заголовков для согласованности с WS-apply.

### 5.2. `internal/app`

- **`app.go`** — сборка зависимостей, регистрация use cases, **инициализация коллаборации** (хаб, Redis, маршруты).
- **`graphql.go`** — настройка GraphQL-сервера; возможная регистрация полей, связанных с collab.
- **`config/`** — структуры конфигурации, в т.ч. **`CollabConfig`** (лимиты WS, Redis URL и т.д.).

### 5.3. `internal/usecase/interactor`

Сценарии уровня приложения: проект, сцена, слои, ассеты, сторителлинг. **`project.go`**, **`project_metadata.go`**, **`scene.go`** — места, где коллаборация может **проверять метаданные** или права перед изменениями.

### 5.4. `internal/infrastructure/mongo` и `memory`

- **Mongo** — основное хранилище документов проектов, сцен, свойств.
- **Memory** — быстрые реализации интерфейсов репозиториев для **юнит-тестов** без Mongo.

### 5.5. `internal/adapter/http`, `internalapi`, `middleware`

Классический HTTP (публикация, файлы, webhooks), gRPC internal API для межсервисного взаимодействия, CORS и прочие middleware Echo.

### 5.6. `pkg/`

Переиспользуемые пакеты: **идентификаторы** (`pkg/id` на базе `reearthx/idx`), модели сцены, property, плагины и т.д. Это **не** «внутренняя кухня» — код, который теоретически можно вынести в библиотеку.

### 5.7. `e2e/` (на Go)

Пакеты тестов `go test ./e2e/...`, которые поднимают или используют реальный сервер/API и Mongo — **тяжёлые**, дольше юнитов, больше открытых соединений к БД.

## 6. Фронтенд `web/` — структура и коллаборация

### 6.1. Общая структура

- **`web/src/app/`** — приложение: **страницы** (`pages/`), **фичи** (`features/`), общие **хуки**, **UI** (`ui/`), утилиты.
- **`web/src/services/`** — «сервисный слой»: **GraphQL** (`gql/` с codegen в `__gen__`), REST, **api/** хуки по доменам (asset, layer, property, …), **i18n**, **routing**, **collab**.
- **`web/src/ee/`** — enterprise/доп. слой (например баннеры) — не ядро OSS-редактора, но в дереве есть.

Редактор картографии живёт в **`app/features/Editor/`** и **`app/features/Visualizer/`**: туда встраиваются панели коллаборации.

### 6.2. Каталог `web/src/services/collab` — файлы

- **`CollabClient.ts`** — низкоуровневый клиент WebSocket: соединение, переподключение, разбор входящих сообщений, отправка исходящих.
- **`CollabProvider.tsx`** — React **Context Provider**: держит состояние коллаборации, подписывает компоненты, связывает с GraphQL/роутингом проекта.
- **`CollabProvider.test.tsx`** — тесты провайдера.
- **`collabContext.ts`** — типы контекста и значения по умолчанию.
- **`useCollab.ts`** — хук для потребления контекста в компонентах.
- **`collabUrl.ts`**, **`collabUrl.test.ts`** — построение URL WebSocket от базового API URL.
- **`applyMessages.ts`** — построение **payload** операций apply (функции экспортируются из `index.ts` длинным списком).
- **`applyMessages.test.tsx`** — тесты сообщений apply.
- **`activityMessages.ts`**, **`activityMessages.test.ts`** — форматы активности.
- **`cursorMessages.ts`**, **`cursorMessages.test.ts`** — форматы курсоров.
- **`chatMessages.ts`**, **`chatMessages.test.ts`** — форматы чата.
- **`chatMentions.ts`**, **`chatMentions.test.ts`** — упоминания в чате на клиенте.

## 8. Технологии: определения, применение, плюсы и минусы

### 8.1. Go (язык сервера)

**Определение:** статически типизированный язык с быстрой компиляцией, встроенной конкуррентностью (goroutines) и простой моделью деплоя (один бинарник).

**Применение здесь:** весь `server/`, включая коллаборацию и e2e-тесты на Go.

**Плюсы:** предсказуемая производительность, сильная стандартная библиотека, простой деплой, хорошая поддержка Mongo и gRPC экосистемой.

**Минусы:** шаблонный бойлерплейт на интерфейсах, меньше «магии» чем в некоторых фреймворках — новичку дольше входить в большой `internal/`.

### 8.2. Echo (HTTP framework)

**Определение:** лёгкий веб-фреймворк для Go.

**Применение:** маршруты REST/GraphQL playground, middleware, WebSocket upgrade для collab.

**Плюсы:** простой API, хорошая интеграция с middleware.

**Минусы:** не «батарейки включены» как у некоторых full-stack фреймворков — много ручной сборки.

### 8.3. gqlgen (GraphQL code generation)

**Определение:** генератор серверного GraphQL на Go из схемы.

**Применение:** типобезопасные резолверы, dataloaders против N+1.

**Плюсы:** строгие типы, скорость разработки API после освоения паттернов.

**Минусы:** цикл «поменял схему → сгенерил → поправил резолверы»; merge конфликты в gen-файлах при неаккуратных PR.

### 8.4. MongoDB

**Определение:** документо-ориентированная NoSQL БД.

**Применение:** основное хранилище проектов, сцен, свойств, ассетов.

**Плюсы:** гибкая схема, хорошо ложится на JSON-подобные структуры сцены.

**Минусы:** сложнее транзакционные сценарии на всю сцену; при тестах много соединений/БД — нагрузка на ОС (см. ulimits).

### 8.5. Redis (опционально для коллаборации)

**Определение:** in-memory data store с pub/sub.

**Применение:** relay сообщений между инстансами, распределённые блокировки.

**Плюсы:** горизонтальное масштабирование реалтайма.

**Минусы:** ещё один компонент в проде; локально без Redis часть функций деградирует до single-instance (что нормально для dev).

### 8.6. Gorilla WebSocket

**Определение:** библиотека WebSocket для Go.

**Применение:** `collab/ws.go`.

**Плюсы:** зрелая реализация протокола.

**Минусы:** нужно внимательно следить за горутинами и backpressure (буферы, лимиты — в коде учтены параметрами конфига).

### 8.7. OpenTelemetry

**Определение:** стандарт наблюдаемости (трейсы/метрики).

**Применение:** инструментирование Echo, Mongo-драйвера и др. (см. `go.mod`).

**Плюсы:** переносимость между бэкендами observability.

**Минусы:** настройка коллекторов может быть нетривиальной для новичка.

### 8.8. React

**Определение:** UI-библиотека с компонентной моделью.

**Применение:** всё дерево `web/src/app`.

**Плюсы:** огромная экосистема, компонентный UX редактора.

**Минусы:** сложность состояния; без дисциплины легко получить лишние ререндеры (частично лечится мemo/context patterns).

### 8.9. TypeScript

**Определение:** надмножество JS со статической типизацией.

**Применение:** весь фронт.

**Плюсы:** раннее обнаружение ошибок, лучший DX в IDE.

**Минусы:** время компиляции, иногда бойлерплейт типов для GraphQL codegen.

### 8.10. Vite

**Определение:** современный dev-server и bundler для фронта.

**Применение:** `yarn start`, production build.

**Плюсы:** очень быстрый cold start dev.

**Минусы:** плагины и native deps иногда требуют тонкой настройки (Cesium — типичный пример).

### 8.11. Vitest

**Определение:** тест-раннер, API совместимый с Jest, нативно дружит с Vite.

**Применение:** `yarn test`, `yarn coverage`.

**Плюсы:** скорость, ESM-first.

**Минусы:** часть Jest-плагинов несовместима напрямую.

### 8.12. Apollo Client (через codegen)

**Определение:** клиент GraphQL с кэшем.

**Применение:** запросы/мутации к серверу, интеграция с React.

**Плюсы:** мощный кэш, нормализация.

**Минусы:** кривая обучения нормализации; коллаборация требует явной синхронизации с WS-событиями.

### 8.13. Cesium

**Определение:** 3D глобус и картографический движок в браузере.

**Применение:** визуализатор сцены.

**Плюсы:** индустриальный стандарт для 3D GIS.

**Минусы:** размер бандла, требования к WebGL и токенам Ion.

### 8.14. Playwright

**Определение:** end-to-end тестирование браузера/API.

**Применение:** каталог `e2e/`, CI workflows `playwright_*.yml`.

**Плюсы:** надёжные селекторы, trace, параллельный запуск.

**Минусы:** медленнее юнитов, хрупкость к таймингам UI без дисциплины ожиданий.

### 8.15. Docker / Docker Compose

**Определение:** контейнеризация и оркестрация локального стенда.

**Применение:** Mongo, fake GCS, accounts stack, dev server.

**Плюсы:** воспроизводимость окружения.

**Минусы:** на Windows без WSL боль; на Linux — права на bind-mount каталоги (`tmp/mongo`).

### 8.16. WebAssembly (упоминание из README)

**Определение:** бинарный формат исполнения в браузере.

**Применение:** выполнение плагинов (см. документацию продукта).

**Плюсы:** изоляция и производительность относительно чистого JS.

**Минусы:** ограничения доступа к DOM, сложность отладки.

### 8.17. OpenID Connect

**Определение:** слой аутентификации поверх OAuth2.

**Применение:** вход пользователей через провайдеров (в dev — mock).

**Плюсы:** индустриальный стандарт.

**Минусы:** настройка redirect URI, секретов, окружений.

## 9. Тесты: как они устроены и как их гонять

### 9.1. Серверные юнит- и интеграционные тесты (Go)

**Команда по умолчанию:** из каталога `server/` выполнить **`make test`**.

Под капотом (см. `server/mk/local.mk`):

1. Проверяется читаемость **`tmp/mongo`** и **`tmp/gcs`** под `server/`: если каталоги существуют, но не читаются текущим пользователем (типичный случай — Docker создал их от root), Makefile **завершится с подсказкой** удалить их через `sudo rm -rf`. Это защита от бессмысленных падений тестов с неочевидной причиной.
2. Запускается  
   `REEARTH_DB=${REEARTH_DB} go test $(if $(TEST_P),-p $(TEST_P),) ${TEST_DIR} -run ${TARGET_TEST}`  
   где по умолчанию `REEARTH_DB=mongodb://localhost`, `TEST_DIR=./...`.

**Параллелизм пакетов:** переменная **`TEST_P`**. Пример: `make test TEST_P=2` ограничивает число параллельно компилируемых/запускаемых пакетов. Это полезно на **слабых машинах** или когда Mongo начинает отваливаться под нагрузкой множества одновременных `go test` пакетов.

**Исключить e2e:** цель **`make test-no-e2e`** — прогоняет все пакеты, кроме `./e2e` (через `go list ./... | grep -v '/e2e$'`). Используется, когда полный `./...` нестабилен локально.

**Отдельно e2e на Go:** `make e2e` → `go test -v ./e2e/...`.

**Отладка:** `make test-debug` пишет verbose лог с таймаутом на короткие пакеты (см. Makefile — полезно точечно, не для всего монорепо сразу).

**Переменные:** `TEST_DIR` можно сузить до `./internal/collab/...` для фокуса.

### 9.2. Почему тесты «ломают» Mongo локально (явно)

Каждый пакет `go test` может:

- поднимать **свою** тестовую БД или коллекции;
- держать **пулы соединений** к `localhost:27017`;
- параллелить подпакеты.

В сумме это даёт **много открытых файловых дескрипторов** на стороне процесса `mongod`. У контейнера Mongo по умолчанию лимит **`nofile`** может быть небольшим → ошибка **Mongo 24 "Too many open files"** → иногда **падение процесса** (в логах Docker бывает код вроде 139) → последующие тесты видят **`connection refused`** или **`server selection timeout`**.

В репозитории в **`docker-compose.yml`** для сервиса **`reearth-mongo`** добавлены **`ulimits.nofile`** (мягкий и жёсткий лимит **65535**), чтобы длинные прогоны (особенно с **e2e**) не убивали базу. После изменения compose контейнер Mongo нужно **пересоздать** (`docker compose up -d` с пересозданием сервиса), чтобы лимиты применились.

### 9.3. Фронтенд-тесты (Vitest)

Из `web/`:

- **`yarn test`** — интерактивный/ watch режим зависит от конфигурации Vitest в проекте.
- **`yarn coverage`** — прогон с покрытием (в CI используется как часть `yarn run coverage`).

Тесты лежат **рядом** с кодом: `*.test.ts`, `*.test.tsx`.

### 9.4. Playwright (`e2e/`)

Скрипты в `e2e/package.json`:

- **`test:api:local`** — API-тесты против локального стенда с mock auth.
- **`test:local`** — UI-тесты против `http://localhost:3000` (фронт должен быть запущен).

Это **отдельный** от Go e2e слой; не путать с `server/e2e`.

### 9.5. CI (GitHub Actions)

- **`ci_server.yml`**: линтер golangci-lint, `make generate`, проверка что **сгенерённый код не отличается** от закоммиченного; job тестов: `go test ./... -race -cover -timeout 10m` с сервисом **mongo:8** на порту 27017 и fake GCS в Docker. Переменная **`REEARTH_DB: mongodb://localhost`**.
- **`ci_web.yml`**: `yarn install`, `yarn type`, `yarn lint`, `yarn coverage`, проверка i18n `yarn run i18n --fail-on-update`, плюс валидация nginx-шаблона.

То есть локально «зелёно» должно быть ближе к CI, если Mongo доступен и лимиты ОС/контейнера адекватны.

## 10. Что было реализовано (итог работ по коллаборации)

Ниже — содержательный список возможностей подсистемы **реалтайм-редактирования**, которые по смыслу выделяются как **реализованный функционал** (сервер + клиент + инфраструктура):

1. **WebSocket-канал** `/api/collab/ws` с параметром проекта, лимитами сообщений и проверкой Origin.
2. **Комнаты по проекту** в in-memory `Hub` с безопасной конкуренцией (`sync.Mutex` / `RWMutex`).
3. **Мультиинстансность** через Redis relay и распределённые блокировки при наличии Redis.
4. **Применение операций (apply)** к сцене: дробление по доменным файлам (слои, стили, виджеты, сторителлинг, свойства, камера и т.д.).
5. **Конфликт-фридли свойства:** LWW часы, HLC для полей, документные часы для merge JSON, сериализация путей apply.
6. **Undo / redo** на сервере с HTTP API и стеком операций.
7. **Аудит применений** с API для выборки истории; панель на клиенте (`CollabApplyHistoryPanel`).
8. **Чат** с историей, упоминаниями и опциональным webhook.
9. **Курсоры и активность** с rate limiting на пользователя.
10. **Блокировки ресурсов** редактора (виджеты и др.) с UX-модалками и гейтами на клиенте.
11. **Версия сцены (scene revision)** для согласования клиента с сервером; GraphQL-связка (`resolver_collab_scene_rev.go`), SSE-вариант в пакете collab.
12. **Клиентский collab-слой:** `CollabProvider`, офлайн-очередь, различение вкладок через `peerInstanceKey`, тосты о чужих apply.
13. **Интеграция GraphQL-мутаций** с коллаборацией в хуках `use*Mutations` — согласованность после персистентных изменений.
14. **Локальная стабильность тестов:** `test-no-e2e`, `TEST_P`, исправление путей томов Mongo, **`ulimits.nofile`** для `reearth-mongo` в compose.

## 11. Заключение

Re:Earth Visualizer — это **большой зрелый продукт** с чётким разделением фронта и бэка и выделенной **подсистемой коллаборации**, которая трогает и транспорт (WS), и домен (apply к сцене), и инфраструктуру (Mongo, Redis), и UX редактора. Тесты многоуровневые: быстрые Vitest на клиенте, обширные Go-тесты на сервере, тяжёлые e2e на Go и Playwright отдельно. Локальная разработка опирается на **Docker Compose в корне** и требует внимания к **правам на `tmp/`** и к **лимитам Mongo**, иначе длинные прогоны будут выглядеть как «флаки», хотя это симптом окружения.

Если этот документ когда-то устареет, самый честный способ его обновить — пройтись по дереву `server/internal/collab` и `web/src/services/collab` и сверить список файлов с репозиторием; имена файлов в Go-проектах почти всегда отражают актуальную декомпозицию.
