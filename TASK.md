Выбор агентного движка

Варианты: Cursor, Codex, Claude code, Opencode. Cursor можно оплачивать по подписке (когда много программируешь, это дешевле), в Opencode можно подключить дешевые китайские модели через Polza.ai без VPN и оплачивать рублями (раньше я пользовался Openrouter, но он хостится на Cloudflare CDN и часто блокируется). На Openrouter.ai в разделе программирование можно посмотреть самые популярные недорогие модели. 

Рекомендации по коду

1) Сделайте ваш проект agent-friendly: создайте в корне файл AGENTS.md, в котором напишите агенту ваш тип ОС, особенности билда и запуска проекта на вашем компьютере, особенности запуска тестов, coding guidelines и инструкции по документированию кода. Впоследствии в этот файл нужно вносить изменения и дополнения, если вы видите, что агент что-то не знает и тратит много времени на то, чтобы это понять.

2) Научитесь запускать тесты. Если тесты отсутствуют, создайте несколько разнообразных функциональных тестов. Не бойтесь просить агента запускать в тестах браузер. Он отлично справляется с созданием тестов, которые запускают в докере сервер, открывают заданный URL в браузере, ждут завершения загрузки страницы и XHR-запросов, проверяют наличие на веб-странице определенных элементов и отсутствие ошибок в консоли браузера.

3) Запустите агента в режиме Plan и попросите исследовать подходы к решению вашей задачи, не внося изменений в код. Пусть он предложит вам несколько вариантов и исследует возможные подводные камни выбранного вами.4) Запустите агента в build для реализации выбранного вами варианта решения задачи

5) Сделайте коммит, чтобы иметь возможность вернуться.

6) Поручите агенту запуск тестов новых функций и исправление возникших ошибок. Если агент не справляется, то ему придется помогать: планировать исправления, просить вставить отладочную печать для понимания причин ошибок, дробить исправления на этапы, делая коммиты, чтобы иметь возможность вернуться назад.

Профессиональным подходом является использование SDD (Spec-Driven Development). Для brownfield-разработки больше всего подходит вариант SDD в реализации OpenSpec. Установите OpenSpec в ваш репозиторий и соответствующий плагин для OpenCode. Перезапустите OpenCode и убедитесь, что у вас появился агент OpenSpec-Plan (третий кроме Plan и Build). Сформируйте план внесения изменений, выполняя последовательно команды: /opsx-explore, /opsx-propose, /opsx-apply, /opsx-verify, /opsx-archive.


Реализация системы совместного редактирования проектов в реальном времени (Real-time Collaboration)

Проект: reearth/reearth-visualizer
Описание задачи

Re:Earth Visualizer — это WebGIS платформа для визуализации геопространственных данных. В настоящее время платформа поддерживает работу нескольких пользователей с одним проектом, но без возможности совместного редактирования в реальном времени. Необходимо реализовать систему real-time collaboration, позволяющую нескольким пользователям одновременно работать над одним проектом, видеть изменения друг друга мгновенно и избегать конфликтов при редактировании.
Функциональные требования
1. Инфраструктура WebSocket соединений

    Реализовать WebSocket сервер на стороне бэкенда (Go) с использованием библиотеки gorilla/websocket или аналогичной
    Создать механизм управления комнатами (rooms) — каждый проект представляет собой отдельную комнату для совместной работы
    Реализовать аутентификацию WebSocket соединений через JWT токены
    Обеспечить масштабируемость — поддержка горизонтального масштабирования через Redis Pub/Sub для синхронизации между серверными инстансами

2. Синхронизация состояния проекта

    Реализовать механизм синхронизации изменений с использованием подхода Operational Transformation (OT) или CRDT для следующих сущностей:
        Слои (layers) — добавление, удаление, изменение свойств
        Стили слоёв (layer styles) — изменение визуальных параметров
        Сцена (scene) — настройки камеры, окружения, освещения
        Виджеты (widgets) — позиционирование и конфигурация
        Storytelling — страницы и блоки историй
    Обеспечить атомарность операций — либо изменение применяется полностью, либо откатывается
    Реализовать буферизацию изменений на клиенте при временных разрывах соединения

3. Визуальные индикаторы присутствия

    Отображение списка активных пользователей, работающих с проектом в данный момент
    Цветовая индикация курсоров других пользователей с их именами/аватарами
    Выделение объектов, которые в данный момент редактируются другими пользователями (блокировка для редактирования)
    Индикаторы «пользователь печатает» или «пользователь перемещает объект»

4. Система блокировок и разрешения конфликтов

    Реализовать механизм оптимистичной блокировки на уровне объектов (layer-level locking)
    Когда пользователь начинает редактирование объекта, другие получают уведомление и видят объект в режиме только для чтения
    Реализовать автоматическое снятие блокировки по таймауту бездействия (например, через 5 минут)
    В случае конфликта (одновременное редактирование) — показывать диалог выбора версии пользователю

5. История изменений и undo/redo

    Расширить существующую систему истории изменений для поддержки многопользовательского режима
    Undo/Redo должны работать только для собственных изменений пользователя
    Визуальное отображение истории с указанием автора каждого изменения
    Возможность отката к любой предыдущей версии проекта администратором проекта

6. Уведомления и коммуникация

    Всплывающие уведомления о действиях других пользователей («Пользователь X добавил слой Y»)
    Встроенный чат для коммуникации между участниками проекта
    Система @упоминаний для привлечения внимания к конкретным изменениям

Технические требования
Бэкенд (Go)

    Расширить GraphQL API новыми мутациями и подписками для real-time обновлений
    Реализовать WebSocket хаб для управления соединениями и бродкастинга сообщений
    Интегрировать Redis для pub/sub между серверными инстансами
    Обеспечить персистентность истории изменений в MongoDB
    Реализовать rate limiting для предотвращения спама в чате

Фронтенд (React + TypeScript)

    Интегрировать WebSocket клиент с существующим Apollo Client для GraphQL subscriptions
    Создать контекст/провайдер для управления состоянием совместной работы
    Реализовать компоненты UI для отображения активных пользователей и их курсоров
    Модифицировать существующие редакторы для поддержки режима только для чтения
    Обеспечить graceful degradation — работа в offline-режиме с последующей синхронизацией

Безопасность

    Валидация всех входящих сообщений WebSocket на стороне сервера
    Проверка прав доступа перед каждой операцией
    Защита от DoS атак через лимиты на размер сообщений и частоту отправки

Ожидаемый результат

По завершении задачи пользователи Re:Earth Visualizer смогут:

    Приглашать других пользователей для совместной работы над проектом
    Видеть изменения, вносимые другими участниками, в реальном времени
    Коммуницировать через встроенный чат
    Избегать конфликтов благодаря системе блокировок
    Отслеживать историю изменений с указанием авторов
    Работать с проектом даже при нестабильном интернет-соединении с последующей синхронизацией

Примечание

Задача требует глубокого понимания архитектуры проекта и затрагивает множество компонентов как на бэкенде, так и на фронтенде. Рекомендуется начать с проектирования протокола коммуникации и API, затем реализовать базовую инфраструктуру WebSocket, и постепенно добавлять функциональность синхронизации для различных типов сущностей.

## Статус реализации в репозитории (оценка, не часть формулировки варианта)

Кодовая база закрывает **ядро MVP и прод‑обвязку** (см. [PLAN.md](PLAN.md), [docs/collab-production-deploy.md](docs/collab-production-deploy.md)): **FR‑1** — WS, комнаты по `projectId`, JWT, Redis pub/sub между инстансами и распределённые блокировки. **FR‑3** — presence, курсоры с **цветной монограммой (avatar chip) и подписью**, typing/move. **FR‑4** — блокировки, TTL, read-only в UI; при конфликте — модалка с **сравнением двух снимков** сцены; ресурс **`style`** для стилей слоя. **FR‑2 (в т.ч. формулировка ТЗ «OT или CRDT»):** операционная синхронизация через **`apply`** + серверную нормализацию; для **Property** — **CRDT LWW‑Register с Hybrid Logical Clock (`fieldHlc` / `propertyFieldHlc`)** и совместимость с целочисленным **`fieldClock`**; **`merge_property_json`** (merge-patch по плоским листьям + **`docClock`** CAS) и **`add_property_item` / `remove_property_item` / `move_property_item`** через collab; для **виджетов** — **`entityClocks`**; для прочих `kind` — опционально **`baseSceneRev`**. Остальное по FR‑2: виджеты (add/update/remove), storytelling (страницы/блоки), **NLS-слои** (add/remove/update/batch), **infobox** на NLS (`create_nls_infobox` / `remove_nls_infobox`; **`useInfoboxMutations(sceneId)`**), **блоки infobox** (`add_nls_infobox_block` / `move_nls_infobox_block` / `remove_nls_infobox_block`; **`useInfoboxBlockMutations(sceneId)`**), **photo overlay** (`create_nls_photo_overlay` / `remove_nls_photo_overlay`; **`usePhotoOverlayMutations(sceneId)`**), **стили слоя** (`add_style` / `update_style` / `remove_style`), **`update_property_value`** с `sceneId` в инспекторе и plugin blocks; при Redis — общие часы/ключи между инстансами; **SSE** и **`subscription { collabSceneRevision }`**, fan-out **sceneRev** по Redis. **FR‑5** — журнал `apply`, REST undo/redo в Mongo + UI (`layerId` / `layerIds`, `userName`, `opKind`, **`GET /api/collab/apply-audit`** с **`sceneId`**); undo/redo для перечисленных `kind` (см. [PLAN.md](PLAN.md)), в т.ч. **`merge_property_json`** (inverse merge-patch), **`add_property_item`** / **`move_property_item`** / **`remove_property_item`** через `undo_exec` (inverse remove / обратный move / inverse **`add_property_item`** со снимком); **админский restore** со снимками Mongo / **501** без коллекции; **redo** после undo remove widget/style. **FR‑6** — чат, @mentions, webhook. **Offline:** `localforage`, flush с хвостом, **WS / online / visibilitychange**. **E2e API:** `e2e/api/tests/collab-ws-two-clients.api.spec.ts` (два WebSocket, `fieldHlc`). **Codegen:** `cd server && make gql` при сети.

По **строгому буквальному** «полный JSON‑CRDT (Automerge/Yjs) по всему дереву property + отдельный dedicated fuzz‑пакет в репозитории» остаётся **инженерный хвост** (оценочно **~5 %**): не все экзотические NLS‑ветки; **undo для `remove_property_item`** реализован через снимок inverse **`add_property_item`** (представительное поле + индекс) и патч redo forward по новому `itemId`. Нагрузочная защита: лимиты сообщений/частоты в hub + **CI** `go test ./...` на сервере (см. `.github/workflows/ci_server.yml`). Для **зачёта по формулировке варианта** (FR‑1–6 и техтребования в объёме brownfield‑MVP) репозиторий **считается закрытым** при наличии AGENTS.md, тестов и деплой‑дока; преподавательские требования п.1–6 из шапки TASK (Plan/OpenSpec/коммиты) — **вне кода**, выполняются студентом.

**Формально зафиксировано в репозитории:** 2026-04-12 — см. раздел «Формальное закрытие варианта (MVP)» в [PLAN.md](PLAN.md).

Остаётся **строго дословно** тяжёлым без отдельного объёма работ: полный документный JSON‑CRDT по всему дереву property (Yjs/Automerge), **фото‑аватары** по URL профиля (сейчас — монограмма + цвет), отдельный **`apply`‑kind под каждое поле «камера / освещение сцены»** если требуется именно отдельный протокольный канал (часть настроек сцены покрывается через **Property** и refetch сцены), отдельный **dedicated fuzz‑target** (вне стандартного `go test`).
