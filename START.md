# START.md — запуск проекта Re:Earth Visualizer для локальной разработки и демонстрации

Этот файл описывает **минимально достаточную последовательность действий** для запуска бэкенда и фронтенда на одной машине. Предполагается ОС **Linux** (или WSL на Windows, как в [README.md](README.md)). Команды указаны так, как их нужно выполнять в терминале: **рабочий каталог** для каждого шага указан явно.

---

## 1. Предварительные требования (обязательно до любых `make` / `yarn`)

| Требование | Проверка | Примечание |
|------------|-----------|------------|
| **Docker Engine** установлен и демон запущен | `docker info` выполняется без ошибки | Бэкенд поднимает MongoDB и вспомогательные сервисы через Compose |
| **Go** | версия из `server/go.mod` (`go` directive) | Нужен для локального `go run` / тестов без Docker-обёртки; для `make d-run` образы уже содержат нужное окружение |
| **Node.js** | версия **не ниже** `20.11.0` | Указано в [web/package.json](web/package.json) (`engines`) |
| **Corepack + Yarn** | `corepack enable` затем активация Yarn 4 | В [AGENTS.md](AGENTS.md) указан Yarn **4.6.0**; в CI может отличаться минор — для локальной работы включите Yarn через Corepack и используйте lockfile репозитория |
| Свободные порты | **8080** — API; **3000** — Vite dev server; **27017** — Mongo при пробросе на хост | Если порт занят, остановите конфликтующий процесс или измените конфигурацию (это уже вне минимального сценария) |

---

## 2. Клонирование и переход в репозиторий

```bash
git clone <URL-вашего-форка-или-upstream> reearth-visualizer
cd reearth-visualizer
```

Дальше все пути **относительно корня** `reearth-visualizer/`, если не указано иное.

---

## 3. Запуск бэкенда (Docker + API)

### 3.1. Первый запуск стенда

1. Перейти в каталог сервера:

   ```bash
   cd server
   ```

2. Поднять все сервисы, которые описаны для цели `d-run` (включая MongoDB, fake GCS, при необходимости accounts — **точный набор см. в выводе** `make help` и в [server/Makefile](server/Makefile) / `server/mk/docker.mk`):

   ```bash
   make d-run
   ```

   **Ожидаемый результат:** команда завершает запуск контейнеров; API становится доступным на **`http://localhost:8080`** (если в вашей конфигурации порт не переопределён).

3. **Один раз** после первого поднятия окружения выполнить инициализацию данных для разработки (mock user, bucket GCS — как задокументировано в [server/README.md](server/README.md)):

   ```bash
   make setup-dev
   ```

   **Ожидаемый результат:** команда завершается с кодом 0; в логах Makefile/README нет инструкции «повторите вручную» без которой вход невозможен.

### 3.2. Повторные запуски

- Если контейнеры уже настроены: снова `cd server && make d-run`.  
- Остановка: `cd server && make d-down` (или цель с полным сбросом — **`make d-destroy`** уничтожает данные; используйте только осознанно, см. `make help`).

### 3.3. Проверка, что API жив

Выполните (из любого каталога):

```bash
curl -s -o /dev/null -w "%{http_code}" http://localhost:8080/api/ping
```

Ожидается HTTP-код **200** (если в проекте нет другого health-path — при отличии смотрите фактический маршрут в [server/internal/app/](server/internal/app/)). Если `curl` не может соединиться — бэкенд не слушает порт 8080: проверьте `docker ps` и логи `make d-run`.

### 3.4. GraphQL Playground (опционально)

При включённой переменной **`REEARTH_WEB_ENABLE_GQL_PLAYGROUND=true`** на стороне web и соответствующей настройке сервера (см. [web/.env.example](web/.env.example)) может быть доступен playground; **точный URL** для вашей сборки смотрите в [AGENTS.md](AGENTS.md) (там указано `GET /graphql` для dev playground сервера). Не смешивайте с `POST /api/graphql` клиента.

## 4. Запуск фронтенда (Vite)

### 4.1. Файл окружения

1. Перейти в каталог web:

   ```bash
   cd web
   ```

2. Создать файл **`.env`** в каталоге `web/` (если его ещё нет). **Рекомендуемый способ:** скопировать шаблон:

   ```bash
   cp .env.example .env
   ```

3. В файле **`web/.env`** обязательно выставить как минимум:

   ```bash
   # API бэкенда (в конце должен быть суффикс /api — как в шаблоне)
   REEARTH_WEB_API=http://localhost:8080/api
   REEARTH_WEB_PLUGINS=http://localhost:8080/plugins
   ```

4. Для локального входа без Auth0 задайте явно:

   ```bash
   REEARTH_WEB_AUTH_PROVIDER=mock
   ```

5. Для отображения Cesium задайте токен (без него часть сцен может не работать):

   ```bash
   REEARTH_WEB_CESIUM_ION_ACCESS_TOKEN=<ваш_токен_из_аккаунта_cesium_com>
   ```

   Ссылка на получение токена — в [README.md](README.md).

### 4.2. Установка зависимостей и старт

Из каталога **`web/`**:

```bash
corepack enable
yarn install
yarn start
```

**Ожидаемый результат:** Vite поднимает dev-server; в консоли указан URL (по умолчанию **`http://localhost:3000`**). Откройте его в браузере **Chromium-семейства** для наименьшего числа сюрпризов с WebGL/Cesium.

---

## 5. Порядок запуска «с нуля» (краткий чеклист)

1. Docker запущен.  
2. `cd server && make d-run` — дождаться готовности.  
3. **Один раз:** `cd server && make setup-dev`.  
4. `cd web` — создать `.env` из `.env.example`, заполнить `REEARTH_WEB_API`, `REEARTH_WEB_PLUGINS`, `REEARTH_WEB_AUTH_PROVIDER=mock`, `REEARTH_WEB_CESIUM_ION_ACCESS_TOKEN`.  
5. `yarn install && yarn start` в `web/`.  
6. Браузер: `http://localhost:3000`.

## 6. Демонстрация collaboration (два клиента)

**Условие:** в редакторе открыт проект с известным **`projectId`** в маршруте/контексте приложения (как в штатном UI Re:Earth после входа в проект). **CollabProvider** подключается при переданном `projectId` в оболочку редактора (см. [web/src/app/features/Editor/index.tsx](web/src/app/features/Editor/index.tsx)).

**Минимальный сценарий:**

1. Запустить бэкенд и фронт по разделам 3–4.  
2. Открыть **два** окна браузера (или два разных браузера) с **одним и тем же** URL проекта/редактора.  
3. Убедиться, что в UI отображается статус collab (полоса presence, см. компонент `CollabPresenceBar`).  
4. В одном окне выполнить изменение, уходящее в `apply` (например свойство слоя / виджет / камера сцены — в зависимости от экрана).  
5. Во втором окне убедиться в обновлении после `sceneRev` / refetch (Apollo `GetScene`).

**Redis для нескольких инстансов API:** если вы поднимаете **более одного** процесса API за балансировщиком, задайте **`REEARTH_COLLAB_REDIS_URL`** (см. [docs/collab-production-deploy.md](docs/collab-production-deploy.md)). Для **одного** локального контейнера API переменная может быть не задана: relay и блокировки будут **in-memory** в процессе.

## 7. Типичные сбои (однозначные причины и действия)

| Симптом | Наиболее вероятная причина | Действие |
|---------|----------------------------|----------|
| `localhost:8080` не отвечает | `make d-run` не выполнен или контейнер упал | `docker ps`, логи контейнеров, повторный `make d-run` |
| Фронт не видит API | Неверный `REEARTH_WEB_API` в `web/.env` | Строка должна быть ровно `http://localhost:8080/api` для стандартного порта |
| 401 / нет пользователя | Не выполнен `make setup-dev` или не `mock` auth | Выполнить `make setup-dev`; выставить `REEARTH_WEB_AUTH_PROVIDER=mock` |
| Пустой глобус / ошибки Cesium | Нет `REEARTH_WEB_CESIUM_ION_ACCESS_TOKEN` | Заполнить токен в `web/.env` |
| Collab не подключается | Нет `projectId` в редакторе или API недоступен | Проверить URL редактора и Network → WS к `/api/collab/ws?...` |
